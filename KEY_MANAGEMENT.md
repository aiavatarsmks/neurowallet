# NeuroWallet — Key Management

> Актуально на 2026-07-04 (после Фазы 0 и Фазы 1). Модель: **non-custodial** —
> весь ключевой материал генерируется, шифруется и используется только в браузере.
> Сервер не видит ни seed, ни ключи, ни пароль кошелька.

## Деривация (единый источник: `frontend/lib/crypto/wallet.ts`)

Один 12-словный BIP39 seed → ключи всех чейнов. Проверяется тестами
с эталонными векторами (`__tests__/crypto/derivation-vectors.test.ts`):
любое случайное изменение деривации валит CI.

| Чейн | Путь | Кривая / метод | Вектор |
|---|---|---|---|
| ETH | `m/44'/60'/0'/0/0` | secp256k1, ethers HDNode | внешний (BIP44) |
| BTC | `m/84'/0'/0'/0/0` → P2WPKH `bc1q…` | secp256k1, ethers + bitcoinjs | внешний (BIP84) |
| TRX | `m/44'/195'/0'/0/0` | secp256k1 | внешний (TronLink) |
| SOL | `m/44'/501'/0'/0'` (все сегменты hardened) | ed25519, SLIP-0010 | внешний (Phantom) |
| TON | `m/44'/607'/0'/0'` (все сегменты hardened) | ed25519, SLIP-0010 | snapshot кодовой базы |

## Шифрование at rest (localStorage)

| Что | Схема | Ключ localStorage |
|---|---|---|
| ETH privkey | ethers keystore JSON, **scrypt N=131072** (2^17) | `wallet_keystore` |
| SOL/BTC/TRX/TON privkey | AES-256-GCM, ключ из **PBKDF2-SHA256, 600 000 итераций**, случайные соль+IV на каждый блоб (`lib/crypto/aes.ts`) | `wallet_sol_enc`, `wallet_btc_enc`, `wallet_tron_enc`, `wallet_ton_enc` |
| Публичные адреса (без ключей) | открытым текстом (нужны для отображения) | `wallet_*_address` |

Каждый чейн шифруется **независимо** (своя соль/IV): компрометация одного
блоба не раскрывает остальные. Пароль один на все блобы.

**Мнемоника не сохраняется никуда и никогда** — живёт только в React-state
во время онбординга (пользователь обязан переписать её и подтвердить 3 слова).

## Unlock-поток и гигиена памяти

1. Пользователь вводит пароль в момент отправки → нужный блоб расшифровывается
   в `Uint8Array` в памяти.
2. Ключ используется для подписи транзакции.
3. Буфер немедленно зануляется `fill(0)` (все call-sites в
   `lib/crypto/transactions.ts` и `wallet.ts`).

Ограничение JS: строковые копии (например, `ethers.Wallet.privateKey`)
занулить нельзя — строки иммутабельны, полагаемся на GC. Полное решение —
Secure Enclave/MPC (Фаза 2, layered custody).

## PIN — слой удобства, не безопасности (`lib/pin.ts`)

PIN (4 цифры) → PBKDF2 → AES-GCM-конверт с **паролем кошелька** →
`wallet_pin_blob`. 5 ошибок → локаут 30 мин (`wallet_pin_attempts`,
`wallet_pin_lockout_until`). Счётчики client-side: злоумышленник с доступом
к localStorage может их сбросить, но это лишь вернёт его к перебору
полноценного пароля против scrypt/PBKDF2 — сами ключи PIN не защищает
и не ослабляет.

## Миграции

### scrypt N=8192 → 131072 (задача 0.4)
Keystore, созданные до 2026-06-24, зашифрованы с N=8192. ethers читает
параметры из самого JSON, поэтому они продолжают работать. При первой
успешной отправке (пароль доказан) `lib/crypto/keystore-migration.ts`
перешифровывает блоб с N=131072 на месте, fire-and-forget. Неверный пароль
не портит блоб; повторная миграция — no-op. Тесты: `keystore-migration.test.ts`.

### XOR-схема (удалена 2026-06-26, задача 0.5)
Исторически SOL/BTC/TRX/TON-ключи хранились как `privkey ⊕ ethPrivkey`
(`wallet_*_xor`): компрометация ETH-keystore раскрывала все чейны. Схема
удалена в пользу независимых AES-GCM-блобов. Ключи и тогда выводились из
мнемоники, поэтому XOR-блобы избыточны — при старте приложение вычищает их
из localStorage (`clearLegacyXorKeys`).

**Последствия для пользователей, создавших кошелёк до 2026-06-26:**
- Отправка SOL/BTC/TRX/TON у них не работает до переимпорта seed
  (ETH работает — keystore не менялся).
- 27 июня менялись пути: BTC `m/44'` P2PKH → `m/84'` SegWit, TON исправлен
  на hardened. После переимпорта адреса BTC/TON будут **новыми**; средства
  на старых адресах текущим приложением недостижимы — если такие есть,
  нужен отдельный recovery-инструмент (обратиться к команде).
  **Важный нюанс:** на своём устройстве legacy-пользователь работает штатно
  (send BTC поддерживает legacy P2PKH через `btc-tx.getSourceType`, TON шлёт
  сохранённым ключом) — проблема возникает только при переимпорте. Полный
  разбор, кто затронут и варианты решения (dual-derive / sweep / warn /
  accept) — в `LEGACY_DERIVATION_JUNE_USERS.md` (открытый custody-вопрос).

## Модель угроз (честные ограничения MVP)

| Угроза | Статус |
|---|---|
| Сервер/БД скомпрометированы | Ключей там нет — терять нечего |
| XSS | Главный риск для localStorage-блобов; митигировано жёстким CSP без `unsafe-inline` (script-src) + `report-uri`. Блобы без пароля бесполезны |
| Кража устройства | Стойкость = стойкость пароля (scrypt 2^17 / PBKDF2 600k). PIN не даёт обхода |
| Вредоносное браузерное расширение | Не митигировано на этом уровне — ограничение web-архитектуры; ответ — Фаза 2 (Secure Enclave / MPC / passkey) |
| Забытый пароль | Восстановление только переимпортом seed — by design. Из Security-center есть кнопка «Забыл пароль → восстановить по seed» (`/onboarding?recover=1`). `verifyWalletPassword` для установки PIN проверяет пароль по любому per-chain блобу, а как fallback — по ETH-keystore (legacy-кошельки без per-chain блобов) |
