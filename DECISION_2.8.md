# DECISION 2.8 — Claim-ссылки «отправил тебе — забери» (design-нота)

> **Статус: на утверждение Максима. Кода нет.** Главная виральная петля Фазы 2
> (паттерн Wallet in Telegram / xRocket). Ниже — разбор развилки удержания
> средств с учётом инвариантов, рекомендация и открытые вопросы.

## Инварианты, которые НЕЛЬЗЯ нарушить (CLAUDE.md)

- **Non-custodial:** ключи только у пользователя; seed/приватный ключ **никогда**
  на backend; сервер **не может** единолично восстановить/двигать средства.
- **Никаких real-money до аудита** — только **demo / test-funds** (testnet).
- **Deny by default**, RLS на всех таблицах, вставки — через service role /
  Edge Functions, не напрямую с клиента; `audit_log` append-only.
- Нейра не имеет доступа к ключам.

## §1. Механика claim — развилка (кто держит средства между «отправил» и «забрал»)

| Вариант | Кто держит средства до claim | Газ | Возврат по expiry | Где состояние | Self-custody? |
|---|---|---|---|---|---|
| **(0) Custodial pool** (как xRocket фактически) | **Backend/сервис** держит средства, ссылка = код | сервис | сервис возвращает | БД сервиса | ❌ **НАРУШАЕТ** non-custodial. Отвергнуто. |
| **(a) Escrow-контракт на TON** | **Смарт-контракт** (trustless, on-chain) | отправитель (депозит) + получатель (claim) | контракт: `withdraw` отправителю после expiry | on-chain + off-chain индекс | ✅ да (управляет код, не сторона). Но нужен **свой аудированный FunC/Tact контракт**. |
| **(b) Ephemeral-key / cheque** | **Эфемерный кошелёк** (свежий адрес, приватный ключ — в ссылке-бэрере) | отправитель фандит эфемерный адрес; claim свипает | отправитель хранит эфемерный ключ локально → свип назад после expiry | off-chain индекс (адрес+статус), **ключ НЕ на backend** | ✅ да (ключ — bearer-секрет в ссылке, backend его не видит). |

**Ключевой момент:** «как xRocket» ≠ non-custodial — xRocket/Wallet cheques
**кастодиальны** (сервис держит средства). Для нас это (0) → отвергнуто.
Non-custodial дают только (a) контракт и (b) эфемерный ключ.

**(b) эфемерный ключ — детали и риск:**
- Отправитель генерит эфемерную TON-пару, фандит адрес суммой, кладёт секрет
  (из которого выводится ключ) в deeplink. Получатель открывает → Mini App
  свипает средства с эфемерного адреса на его (новый/существующий) кошелёк.
- Backend хранит только **не-чувствительное**: эфемерный **публичный адрес**,
  статус, expiry, сумма, coin. **Приватного ключа на backend нет** — инвариант цел.
- Риск bearer-инструмента: ключ едет через Telegram (ссылка в чате) — кто
  перехватил ссылку, тот и заберёт. Это by design (отправитель шлёт нужному
  получателю), но Telegram технически видит ссылку. **Митигируется тем, что v1
  — только testnet/demo** (мелкие тестовые суммы), expiry, лимиты.

**(a) escrow-контракт — детали:**
- Секрет не кладётся в ссылку как ключ; claim — доказательство знания секрета
  контракту. Ключ отправителя не покидает его кошелёк. Самый trustless.
- Требует **написать и проаудировать TON-контракт** — тяжело, и упирается в
  «no real money до аудита». Целевой вариант для mainnet-real-money.

## Рекомендация — фазировать

1. **v1 = DEMO-петля (симуляция, без цепочки и без custody).** Claim-ссылка в
   demo создаёт симулированный claim, «зачисляющий» demo-баланс получателю.
   Цель — доказать **виральную петлю + аналитику активации** (link → open →
   онбординг → claim) с **нулевым** custody/chain-риском. Полностью в рамках
   «только demo». Deeplink-контур и UX — настоящие.
2. **v2 = testnet, механика (b) эфемерный ключ.** Тот же UX, реальные testnet-TON.
   Проверяем on-chain свип/возврат. Всё ещё «test funds» → инвариант цел.
3. **v3 = mainnet, механика (a) escrow-контракт, ТОЛЬКО после аудита** (контракта
   и общего security-аудита Фазы 3). Real money.

Deeplink-контур и схема данных проектируются так, чтобы механику удержания
можно было заменить (симуляция → эфемерный → контракт) **без переписывания UX**.

**Почему так:** максимум обучения (виральность/конверсия) при нулевом нарушении
инвариантов; реальные деньги — строго после аудита; ни в одном варианте
приватный ключ не попадает на backend, сервер не держит средства единолично.

## §2. TON-first

- Дефолт claim = **TON / USDT-TON** (эфемерный адрес — TON-кошелёк). Ложится на
  Receive=TON и общий TON-native акцент (2.10).
- Связь с TON DNS резолвером (2.6): claim-ссылка несёт секрет, **DNS-резолв не
  нужен** для самого claim. Но «request money» (обратная петля) может
  использовать TON DNS / @username как получателя — там резолвер переиспользуется.

## §3. Анти-фрод и границы

- **Expiry** (напр. 72ч / 7 дней — вопрос к тебе); по истечении — возврат отправителю.
- **Ровно одна неистёкшая ссылка на перевод** (идемпотентность: dedupe-ключ на
  (sender, transfer) — как в notifications 2.4).
- **Лимиты**: сумма на ссылку, число активных ссылок / частота (rate-limit как в
  api-security).
- **Возврат отправителю**: v1 — вернуть demo-баланс; v2 — свип эфемерного адреса
  назад (ключ у отправителя локально).
- **demo↔real граница (инвариант):** из demo — **только симулированные** claim,
  ни одного реального/testnet-действия. Явный гейт `isDemo`.
- **Двойной claim**: атомарная смена статуса `funded → claimed` (уникальность,
  compare-and-set) — нельзя забрать дважды.

## §4. Схема данных (proposal)

- **`claim_links`**: `id`, `sender_user_id`, `ephemeral_address` (публичный, не ключ),
  `coin`, `amount`, `status` (`created|funded|claimed|expired|refunded`),
  `dedupe_key` (uniq на активную ссылку), `expires_at`, `claimed_by_user_id?`,
  `created_at`. **Приватного ключа НЕТ.** RLS: отправитель видит свои; **claim-lookup
  по ссылке — через server-route (service role)**, возвращает только не-чувствительный
  статус по `ephemeral_address` из ссылки (получатель не владеет строкой).
  Смена статуса — только service role (атомарно).
- **`claim_events`** (или переиспользовать `analytics_events` + `audit_log`):
  append-only лог `created/opened/claimed/refunded` с `trace_id`.
- **`audit_log`**: `claim_created` / `claim_claimed` / `claim_refunded` (service role,
  append-only) — без сумм-в-тексте сверх необходимого, без ключей.

## §5. Аналитика (цель 15–35% активаций из p2p)

События: `claim_link_created`, `claim_link_opened`, `wallet_created_from_claim`,
`claim_completed` (+ `claim_refunded`). Формулы:
- **Активация из p2p** = `wallet_created_from_claim / claim_link_opened` (новые
  пользователи, пришедшие по claim и создавшие кошелёк).
- **Конверсия ссылки** = `claim_completed / claim_link_created`.
- Склейка анонимного `claim_link_opened` (pre-auth) с юзером — через
  `session_identified` (механизм 1.1 уже есть).

## §6. Feature flag, тесты, deeplink

- **Feature flag**: env/config `CLAIM_LINKS_ENABLED` (по умолчанию off) — включаем
  постадийно.
- **Deeplink-контур**: `https://t.me/NeuroWallet_bot/app?startapp=<claim_ref>` →
  Mini App парсит `startapp` → экран claim. **В `startapp` НЕ кладём приватный
  ключ v2** (виден Telegram/в ссылке): для v2 секрет — во **fragment** ссылки
  (`#...`, не уходит на сервер) или отдельный слой; для v1-demo секрет не нужен
  (claim_ref → server статус). Это отдельный под-вопрос к проработке в v2.
- **Тесты**: генерация/кодирование-декодирование claim_ref; expiry/refund логика;
  запрет двойного claim (атомарность); dedupe одной активной ссылки; demo-гейт
  (из demo нет реальных действий); RLS/route для claim-lookup; аналитические
  события. Цель — как обычно, зелёный CI без падения текущих 157.
