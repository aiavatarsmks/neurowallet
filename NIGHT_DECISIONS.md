# NIGHT_DECISIONS — автономная ночная сессия 2026-07-02/03

> Решения, которые обычно согласовывались бы с Максимом. Формат:
> решение / альтернатива / почему так.

## Задача 1.3 (risk scoring + anti-poisoning)

### D-1.3-1: Таблиц две, а не четыре
**Решение:** миграция 0004 создаёт `risk_events` и `override_actions`.
`recipient_profiles` и `poisoning_alerts` из плана не создаю.
**Альтернатива:** все четыре таблицы сразу.
**Почему:** recipient-профили выводимы из `tx_drafts` (distinct to_address,
статусы), poisoning-alert — это `risk_events` с reason `poisoning_similarity`.
Отдельные таблицы = дублирование данных без нового сигнала. Консервативно:
меньше поверхности, добавить таблицы потом дешевле, чем выпиливать.

### D-1.3-2: Адресов нет ни в risk_events, ни в аналитике
**Решение:** risk_events хранит только coin, level, reasons (коды), trace_id,
draft_id. Сам адрес — только в `tx_drafts` (политика из 1.2).
**Альтернатива:** класть адрес в risk_events для удобства анализа.
**Почему:** правило Максима «финансовое поведение в одном месте» (см.
SUPABASE_SCHEMA.md, privacy-блок). Джойн по draft_id/trace_id даёт то же.

### D-1.3-3: Blocklist поставляется пустым
**Решение:** `lib/risk/blocklist.ts` — структура + пустой seed, тесты на
механику через инжектируемые записи.
**Альтернатива:** захардкодить «известные скам-адреса» из интернета.
**Почему:** непроверенный список = риск ложных блоков реальных адресов.
Наполнение списка — осознанное действие владельца продукта, не ночное
решение агента. Vendor intel (Blockaid/TRM) — Фаза 2 по плану.

### D-1.3-4: Разные политики override для block-причин
**Решение:** `blocklisted` — жёсткий блок без override; `poisoning_similarity`
— override возможен через явный чекбокс «Я понимаю риск» + запись в
override_actions + audit_log (`risk_override_confirmed`) + пароль кошелька
(он и так обязателен для отправки — это фактический step-up фактор MVP).
**Альтернатива:** единая политика для всех block-причин.
**Почему:** санкционный/скам-список не должен обходиться из UI вообще;
poisoning-эвристика вероятностная — легитимный новый адрес может быть похож
на старый, полный запрет = ложные отказы. Биометрии в web-MVP нет, пароль —
максимальный доступный step-up (усиление — Фаза 1.6/2.1).

## Задача 1.4 (address book + NeuroID)

### D-1.4-1: recipient_history и alias_claims таблицами не создаю
**Решение:** из четырёх таблиц плана создаю только `contacts` (0005).
**Альтернатива:** все четыре.
**Почему:** recipient_history уже фактически существует (представление над
`tx_drafts` через /api/recipient-history из 1.3); alias_claims — это
существующая `neuro_directory` (детерминированный id из uuid + UNIQUE =
anti-collision уже гарантирован БД). Дублировать — плодить рассинхрон.

### D-1.4-2: Picker не переписываю — расширяю существующий SendScreen
**Решение:** favorites (звёздочка + сортировка), recent recipients (чипы
под полем адреса из /api/recipient-history), серверная синхронизация
localStorage-контактов (однократный аплоад, сервер — источник истины).
**Альтернатива:** новый полноценный recipient-picker компонент.
**Почему:** SendScreen уже реализует «контакты / NeuroID / адрес в одном
селекторе» — ночная переделка UI без владельца продукта = риск. Деградация
до миграции 0005: GET пустой, POST 503, локальная копия работает как раньше.

### D-1.4-3: Privacy-настройки NeuroID отложены
**Решение:** не делаю toggle видимости в директории этой ночью.
**Альтернатива:** колонка is_public + фильтр в resolve.
**Почему:** продуктовое решение о дефолте видимости (opt-in vs opt-out)
должно быть за Максимом — оно меняет UX обещание существующим тестерам.
Добавлено в MORNING_CHECKLIST как вопрос.

## Задача 1.5 (QR + paylinks)

### D-1.5-1: «Подписанные ссылки» = uuid-capability + серверный резолв, без HMAC
**Решение:** ссылка вида /pay/<uuid>; резолв только через
/api/payment-request (service role по точному id, rate limit по IP);
у anon-роли НЕТ политик на таблицу.
**Альтернатива:** HMAC-подпись параметров в самой ссылке.
**Почему:** подпись защищает от подделки параметров, но здесь параметры
не в ссылке — они в БД, а ссылка лишь непредсказуемый ключ (128 бит).
Подделать нечего; перебор uuid невозможен; expiry enforce'ится сервером.
HMAC добавил бы код и ротацию секрета без нового свойства безопасности.

### D-1.5-2: QR платёжной страницы кодирует адрес, а не EIP-681/deep-link
**Решение:** QR на /pay/<id> содержит голый адрес получателя.
**Альтернатива:** чейн-специфичные URI (ethereum:, ton://…).
**Почему:** плательщик может пользоваться любым кошельком; голый адрес
сканируется всеми, URI-схемы поддерживаются частично и различаются по
чейнам. Сумма показана текстом на странице.

## Задача 1.6 (security center lite)

### D-1.6-1: Секция в профиле вместо отдельного экрана
**Решение:** SecurityCenter — компонент внутри ProfileScreen.
**Альтернатива:** новый таб/экран.
**Почему:** объём контента (PIN-статус, устройства, фид, revoke) — одна
карточка; новый экран = навигация и роутинг ночью без владельца продукта.

### D-1.6-2: «История экспорта ключей» и step-up перед экспортом seed — N/A
**Решение:** пункты плана пропущены как беспредметные.
**Почему:** в приложении НЕТ экспорта seed/ключей — мнемоника вообще не
хранится (см. KEY_MANAGEMENT.md). Step-up перед override уже сделан в 1.3.
Когда появится экспорт — тогда и step-up с историей.

### D-1.6-3: Revoke = supabase signOut(scope:'others')
**Решение:** «Выйти на всех других устройствах» через встроенный механизм
GoTrue; таблица device_sessions не создаётся, только devices (heartbeat).
**Альтернатива:** самодельный реестр сессий с точечным revoke.
**Почему:** GoTrue-revoke работает мгновенно и надёжно (refresh-токены
инвалидируются сервером Supabase); самодельный реестр не может отозвать
JWT до его истечения (TTL 900 c) и лишь имитировал бы точечность.

### D-1.3-5: Первая отправка на адрес = warning, не block
**Решение:** first_seen — жёлтый shield с баннером, кнопка активна.
**Альтернатива:** блокировать до подтверждения.
**Почему:** у ВСЕХ пользователей до применения миграции 0003 история пуста —
каждая отправка была бы «первой» и блокировалась бы. Warning честно
отражает уверенность эвристики.
